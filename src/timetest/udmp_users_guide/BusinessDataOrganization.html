1.2 业务数据组织方式

业务数据的组织方式

为应对不同业务系统的灵活需求，UDMP 提供了一个定义较为宽松，弹性较大的业务数据组织方式。

数据归属

接入 UDMP 的业务系统称为应用（App）。在 UDMP 里存储的任何数据一定会明确归属于某个 App。一个 App 对归属于自己的数据拥有包括增删查改等完整的控制权。

文档

所有的业务数据应该以文档（Document）为单位进行组织。文档在概念上可以被理解为一个文件包，或者一个文件夹。一个文档内主要包含以下数据：

  • 任意数量（至少一个）的文档页。
  • 任意数量（至少一个）的结构化业务数据（Bizdata）字段。
  • 一个文档 ID（DocId） 字符串。

如果有必要，App 可以通过替换一个文档中的特定文档页对其进行修订，也可以向一个文档中继续追加新的文档页。

文档页

文档页是直接容纳非结构化数据文件的基本对象。一个文档页（Page）内可以包含以下数据：

  • 一个非结构化业务数据文件。
  • 任意数量的结构化业务数据（Bizdata）字段。
  • 一个文档页码（PageNo）数字。

同一个文档中的文档页可以容纳完全不同的非结构化数据文件，例如第一页是视频，第二页是图片，第三页是 PDF 文件。

同一个文档中的文档页在存储或是被查询返回时都会始终保持有序，顺序由上传文档时添加文档页的顺序决定。PageNo 字段就是对此顺序的直观表达。

字段解释

  1. DocId 用于在 UDMP 整个系统中唯一标识一个 Document，它是一个长度 80 字节的字符串，在上传文档时由 UDMP API 自动生成。
  2. PageNo 用于在文档内唯一标识一个文档页。Java SDK 在上传文档时按文档页添加顺序，从 1 开始递增生成 PageNo。因此一个文档中最大的 PageNo 与文档页数量在数字上完全相同。
  3. Bizdata 字段可以由业务系统自行命名，字段值仅支持字符串数据类型。一个 Document 或 Page 可以有无限数量个 Bizdata 字段。
