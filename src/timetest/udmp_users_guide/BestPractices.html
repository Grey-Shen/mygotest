4. 最佳实践

最佳实践

文档的结构化业务数据

文档必须拥有自己的 Bizdata 字段。例如：一笔开户的交易，柜面操作的时候产生了 3 张图片。那么将数据上传到 UDMP 时，需要先初始化一个文档，这个文档的属性可以有业务流水号，有账号，有客户号，也可以有其它任何业务数据。

设计 Bizdata 字段的主要业务目的为：

  1. 减少不必要的数据流转。例如：业务系统 A 上传了一个文档，其中附带业务流水号。那么其它的业务系统就可以根据相同的业务流水号直接查询到该文档，无需再从业务系统 A 采集文档 ID 信息。
  2. 构建以客户中心的多维度非结构化数据集。例如：客户进行信用卡开户，或是在柜面开通借记卡均会提供身份证号。那么仅使用身份证号作为查询条件，就可以访问某个具体客户的非结构化数据集。

因此业务系统在初始化文档时，应该确保 Bizdata 字段数量至少大于 1 个。

Bizdata 字段使用规范

使用 Bizdata 字段应该遵循以下基础规范：

  1. 必须符合 MongoDB 的 field 命名规范，不能含有英文的 .、$以及任何空白字符。参考 MongoDB 官方文档。
  2. 必须符合我行数据质量管控组的要求。参考我行数据库字段命名规范。
  3. 业务系统设计好 Bizdata 字段后，请将字段命名发送给数据质量管控员和 UDMP 开发团队，进行复核确认。

文档页的结构化业务数据

与文档一样，文档内的每一个文档页也都可以拥有自己的 Bizdata 字段。例如名称，类型等。文档页的 Bizdata 字段使用仍需遵循如上规范。但与文档不同，文档页可以没有任何 Bizdata 字段。

文档和文档页设计考量

  1. 设计文档的时候应该从业务数据使用场景的角度考虑，尽量将一些比较重要的基本数据加到文档的 Bizdata 字段中。例如：业务流水号、BECIF 号、账号等。
  2. 从交易的角度看，一笔交易可能在不同的流程阶段产生不同的非结构化数据文件。在设计上应该通过追加的方式将新产生的文档页追加到这笔交易已有的文档中，尽可能避免一笔交易对应多个文档。
